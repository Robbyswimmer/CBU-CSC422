# Track 1: Final Project Rubric

## Overview

**Total Points: 28 points**
- Final Project Proposal: 2 points (already graded)
- Final Project Implementation: 26 points (breakdown below)

## Detailed Scoring Rubric

### Repository Quality (8 points total)

#### Code Organization & Quality (3 points)

| Points | Criteria | Description |
|--------|----------|-------------|
| **3 - Excellent** | Professional Code Quality | Code is clean, modular, and follows PEP 8 standards. Functions and classes are well-designed with clear interfaces. Variable names are descriptive. Complex logic is broken into manageable functions. Error handling is appropriate. |
| **2 - Proficient** | Good Code Quality | Code is mostly clean and organized with minor style issues. Most functions are well-designed. Variable names are generally descriptive. Some opportunities for better modularization. |
| **1 - Developing** | Basic Code Quality | Code works but has style inconsistencies. Some functions are overly complex or poorly named. Limited modularization. Some code duplication. |
| **0 - Inadequate** | Poor Code Quality | Code is difficult to read and understand. Poor naming conventions. Significant style violations. Monolithic structure with little organization. |

#### Reproducibility & Documentation (2 points)

| Points | Criteria | Description |
|--------|----------|-------------|
| **2 - Excellent** | Fully Reproducible | Complete requirements.txt/environment.yml with version specifications. Clear setup instructions that work from clean environment. All random seeds set. Data acquisition scripts provided or clear instructions. Code runs without errors. |
| **1.5 - Proficient** | Mostly Reproducible | Good dependency management with minor omissions. Setup instructions are mostly clear. Most randomness controlled. Minor issues that can be easily resolved. |
| **1 - Developing** | Partially Reproducible | Basic dependency information provided. Some setup instructions present but may be incomplete. Some reproducibility issues that require troubleshooting. |
| **0.5 - Inadequate** | Difficult to Reproduce | Missing or incomplete dependency information. Unclear setup process. Significant barriers to reproduction. |
| **0 - Unacceptable** | Cannot Reproduce | No dependency management. Code cannot be run by others. Major reproducibility failures. |

#### Professional Presentation (2 points)

| Points | Criteria | Description |
|--------|----------|-------------|
| **2 - Excellent** | Professional Repository | Comprehensive README with clear project description, setup instructions, usage examples, and results summary. Well-organized directory structure. Meaningful commit history with descriptive messages. Professional presentation suitable for portfolio. |
| **1.5 - Proficient** | Good Repository | Good README with most essential information. Reasonable directory structure. Adequate commit history. Generally professional appearance. |
| **1 - Developing** | Basic Repository | Basic README with some information. Some organization present. Limited commit history. Adequate but not polished. |
| **0.5 - Inadequate** | Poor Repository | Minimal or unclear README. Poor organization. Sparse commit history. Unprofessional presentation. |
| **0 - Unacceptable** | Inadequate Repository | Missing or severely inadequate documentation. No clear organization. Poor version control practices. |

#### Technical Implementation (1 point)

| Points | Criteria | Description |
|--------|----------|-------------|
| **1 - Excellent** | Correct Framework Usage | Proper use of ML/DL frameworks with appropriate APIs. Follows best practices for chosen tools. Efficient implementation with good performance. |
| **0.75 - Proficient** | Good Framework Usage | Generally correct use of frameworks with minor inefficiencies. Mostly follows best practices. |
| **0.5 - Developing** | Basic Framework Usage | Basic use of frameworks with some incorrect patterns. Some inefficiencies or misuse of APIs. |
| **0.25 - Inadequate** | Poor Framework Usage | Significant misuse of frameworks. Inefficient or incorrect implementations. |
| **0 - Unacceptable** | Framework Misuse | Major framework misuse or failure to use appropriate tools. |

## Late Submission Policy

- **Late submissions** will be penalized according to course policy

- **Technical difficulties** should be reported immediately to instructor

## Academic Integrity

This project must represent your individual work. While you may:
- Use standard ML/DL libraries and frameworks
- Reference and cite external resources appropriately
- Discuss general concepts with classmates

You may NOT:
- Copy code from other students or online sources without attribution
- Collaborate on implementation or writing
- Submit work that is substantially similar to another student's project
- Use AI-generated code without disclosure and understanding
